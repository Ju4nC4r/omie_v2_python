# Capítulo 1. Resumen

## Predicción del mercado eléctrico español mediante técnicas de aprendizaje automático

### 1.1. Contexto del mercado eléctrico español

El mercado eléctrico español constituye uno de los elementos fundamentales del sistema energético nacional. En el se coordinan diariamente generadores, comercializadoras, consumidores, operadores del sistema y operadores del mercado con el objetivo de garantizar que la energía eléctrica demandada pueda ser suministrada de forma segura, eficiente y económicamente viable. La electricidad presenta una característica especialmente relevante frente a otros bienes: debe producirse y consumirse prácticamente en tiempo real, ya que el almacenamiento a gran escala sigue siendo limitado y costoso. Esta particularidad convierte la planificación, la predicción y la operación del sistema en tareas de gran complejidad técnica.

En España, el mercado diario de electricidad está gestionado por OMIE, el Operador del Mercado Ibérico de Energía. En este mercado se negocia la energía eléctrica para cada periodo del día siguiente. Los agentes presentan ofertas de compra y venta, y el sistema determina un precio marginal para cada periodo. Este precio es especialmente importante porque sirve como referencia para buena parte de la actividad económica asociada al sector eléctrico. Aunque existen contratos bilaterales, coberturas financieras y mercados intradiarios, el precio del mercado diario sigue siendo un indicador central para analizar la evolución del coste de la energía.

La formación del precio eléctrico depende de numerosos factores. Entre ellos destacan la demanda prevista, la disponibilidad de generación, el coste de combustibles como el gas natural, el precio de los derechos de emision de CO2, las restricciones técnicas, el estado de las interconexiones internacionales y, de forma cada vez más relevante, la producción renovable. La penetración creciente de tecnologías como la eólica y la solar fotovoltaica ha modificado de forma significativa la dinámica del mercado. En horas de alta generación renovable, especialmente solar, los precios pueden caer de forma notable. En cambio, cuando la producción renovable es baja y la demanda es elevada, tecnologías con costes variables superiores pueden marcar precio.

Este nuevo escenario introduce oportunidades y retos. Por un lado, la integración de renovables permite reducir emisiones y disminuir la dependencia de combustibles fósiles. Por otro, aumenta la variabilidad del sistema y hace que los precios sean más sensibles a las condiciones meteorológicas. La predicción del precio eléctrico se convierte así en un problema de interés tanto técnico como económico. Comercializadoras, generadores, consumidores industriales, agregadores de demanda y gestores de almacenamiento pueden beneficiarse de estimaciones más precisas del precio futuro.

Desde el punto de vista académico, la predicción del precio eléctrico es un caso de estudio adecuado para aplicar técnicas de series temporales y aprendizaje automático. Combina datos históricos, variables exógenas, patrones cíclicos, estacionalidad, ruido, eventos extremos y cambios de régimen. Por ello, representa un problema suficientemente rico para evaluar distintos enfoques de modelado, desde métodos lineales hasta redes neuronales y modelos basados en árboles.

En este Trabajo Fin de Grado se propone estudiar el problema de la predicción del precio del mercado eléctrico español mediante técnicas de aprendizaje automático, utilizando datos públicos de OMIE y, de forma opcional, datos de previsión de generación renovable procedentes de ESIOS/REE. El trabajo incluye tanto una parte teórica y metodológica como una aplicación práctica desarrollada en Python, orientada a descargar datos, preparar variables, entrenar modelos, comparar resultados y realizar inferencia.

La relevancia de este estudio se justifica también por la creciente digitalizacion del sector energético. La disponibilidad de datos públicos, APIs, históricos de mercado y herramientas de análisis permite desarrollar prototipos capaces de aproximarse a problemas que hace años requerian infraestructuras más complejas. Desde una perspectiva formativa, esto permite conectar conocimientos de programación, estadistica, sistemas eléctricos y aprendizaje automático en un único proyecto aplicado.

El mercado eléctrico, además, presenta una dificultad adicional: sus precios no evolucionan de manera suave ni completamente predecible. Pueden aparecer precios muy bajos en situaciones de exceso de oferta renovable, precios elevados por tensiónes de demanda o combustible, y comportamientos atípicos durante festivos, fines de semana o episodios meteorológicos extremos. Por tanto, el objetivo no es encontrar una formula determinista, sino construir modelos capaces de aprender patrones útiles a partir de los datos disponibles.

Este contexto convierte la predicción del precio eléctrico en un problema especialmente adecuado para un Proyecto de carácter técnico. Permite trabajar con datos reales, construir un sistema software completo y evaluar los resultados de forma cuantitativa. Al mismo tiempo, obliga a reflexionar sobre las limitaciones de los modelos, la importancia de la calidad de los datos y la necesidad de evitar interpretaciones excesivamente optimistas de las métricas.

### 1.2. Objetivo principal del trabajo

El objetivo principal de este trabajo es desarrollar y evaluar un sistema capaz de estimar el precio del mercado eléctrico español a partir de datos históricos y variables relacionadas con la generación renovable. El sistema se plantea como una herramienta experimental y educativa, no como una plataforma de predicción comercial. Su finalidad es demostrar como puede construirse un flujo completo de aprendizaje automático aplicado a un problema real del sector energético.

El proyecto se centra en el precio marginal español del mercado diario publicado por OMIE. Este precio se identifica en los ficheros `MARGINALPDBC` mediante el campo `marginal_es`, expresado en euros por megavatio hora. La predicción se formula como un problema supervisado: a partir de información disponible en periodos anteriores y de variables temporales, el modelo intenta estimar el precio de un periodo posterior.

Para alcanzar este objetivo se definen varias metas concretas. En primer lugar, se requiere construir un mecanismo automatizado de descarga y parseo de datos OMIE. En segundo lugar, se necesita transformar esos datos en una estructura adecuada para el entrenamiento de modelos, incorporando retardos, medias móviles, variables cíclicas de calendario y otros indicadores derivados. En tercer lugar, se deben entrenar y comparar distintos modelos de aprendizaje automático. Finalmente, se debe construir una aplicación práctica que permita ejecutar el flujo completo de manera sencilla, tanto desde consola como desde una interfaz gráfica.

El trabajo también busca analizar la utilidad de incorporar variables externas relacionadas con la producción renovable. Para ello se contempla la integración opcional con ESIOS, la plataforma de información del sistema eléctrico gestionada por Red Eléctrica. En concreto, se consideran indicadores de previsión eólica y solar. La hipótesis de partida es que estas variables pueden aportar información relevante, ya que la disponibilidad renovable influye directamente en la formación del precio eléctrico.

Otro objetivo importante es comparar modelos de diferente naturaleza. No se pretende asumir de antemano que una red neuronal será necesariamente superior, sino evaluar varios enfoques bajo una misma metodología. Por este motivo, la aplicación permite entrenar una regresión Ridge, una red neuronal multicapa y un modelo de gradient boosting. Además, incluye un modo automático que selecciona el modelo con mejor comportamiento según la métrica MAE en validación temporal.

La inclusión de una aplicación práctica persigue acercar el trabajo a un caso de usó real. El usuario puede elegir fechas, descargar datos, preparar el dataset, seleccionar modelo, entrenar, evaluar y generar una predicción. Esta aplicación constituye el núcleo experimental del Proyecto y sirve como base para analizar los resultados obtenidos.

Como objetivo transversal, se busca que el sistema sea modular. Cada parte del flujo se implementa en un módulo independiente: descarga de datos, integración con ESIOS, generación de variables, entrenamiento, inferencia e interfaz gráfica. Esta separación facilita la mantenibilidad y permite ampliar el proyecto en el futuro. Por ejemplo, se podrían incorporar nuevas fuentes de datos sin modificar por completo el entrenamiento, o se podrían añadir nuevos modelos sin alterar el módulo de descarga.

También se pretende que el trabajo sea reproducible. Para ello se documentan los comandos de instalación, se define un entorno virtual, se fija una estructura clara de carpetas y se excluyen de Git los artefactos generados, como datasets descargados, modelos serializados y caches. La reproducibilidad es especialmente importante en proyectos de aprendizaje automático, ya que pequeñas variaciones en datos, parámetros o particiones temporales pueden alterar los resultados.

Finalmente, el trabajo tiene un objetivo pedagogico: mostrar de forma comprensible el recorrido que va desde un dato bruto publicado por un operador de mercado hasta una predicción generada por un modelo entrenado. Este recorrido incluye decisiones prácticas que suelen ser tan relevantes como el algoritmo elegido: como parsear datos, como alinear timestamps, como tratar valores ausentes, como construir retardos y como evaluar sin mezclar futuro y pasado.

### 1.3. Metodología empleada

La metodología del trabajo se organiza en varias fases consecutivas. La primera fase consiste en la obtención de datos. Para el precio eléctrico se utilizan ficheros públicos de OMIE. Estos ficheros contienen información diaria del precio marginal para España y Portugal. El programa descarga los ficheros correspondientes al rango de fechas seleccionado y los almacena localmente en una carpeta de cache. Esta estrategia evita repetir descargas innecesarias y facilita la reproducibilidad de los experimentos.

La segunda fase consiste en el procesamiento de los datos. Los ficheros OMIE se parsean y se transforman en un dataframe estructurado con columnas como fecha, periodo, timestamp, precio marginal portugués y precio marginal español. Dado que el mercado puede presentar datos horarios o cuarto-horarios, el sistema está preparado para detectar si un día contiene 24 o 96 periodos y asignar correctamente el timestamp correspondiente a cada observación.

La tercera fase es la ingeniería de variables. En problemas de predicción de series temporales, la calidad de las variables construidas suele ser tan importante como el modelo empleado. En este proyecto se generan variables de calendario, como hora, día de la semana, mes e indicador de fin de semana. También se crean variables cíclicas mediante transformaciones seno y coseno, lo que permite representar correctamente magnitudes periodicas. Por ejemplo, las 23:00 y las 00:00 están proximás en el ciclo diario, aunque numéricamente 23 y 0 parezcan alejadas.

Además, se incorporan retardos del precio. Estos retardos permiten que el modelo observe precios pasados relevantes: el periodo anterior, varias horas previas, el día anterior, la semana anterior y otros puntos temporales. También se calculan medias móviles, desviaciones móviles, mínimos, máximos, diferencias y ratios. Estas variables resumen el comportamiento reciente del mercado y ayudan al modelo a capturar tendencias, volatilidad y patrones repetitivos.

La cuarta fase introduce variables externas opcionales procedentes de ESIOS. El programa puede descargar previsiones de generación eólica, solar fotovoltaica y solar térmica. Estas variables se unen al dataset de OMIE por timestamp y se transforman en columnas adicionales, como generación renovable prevista total o ratio entre eólica y solar. Esta parte requiere un token de acceso a la API de ESIOS, por lo que se ha implementado como funcionalidad opcional.

La quinta fase consiste en la división temporal del dataset. En lugar de mezclar aleatoriamente las observaciones, se utiliza una separación cronológica. Los primeros datos se emplean para entrenamiento y los últimos para validación. Esta decisión es esencial para evitar fuga de información futura. En un problema temporal, no sería realista entrenar con datos posteriores al periodo que se quiere predecir.

La sexta fase corresponde al entrenamiento de modelos. El sistema implementa tres candidatos principales. El primero es `RidgeCV`, una regresión lineal regularizada que sirve como modelo rápido, estable y competitivo. El segundo es `MLPRegressor`, una red neuronal multicapa sencilla que permite capturar relaciones no lineales. El tercero es `HistGradientBoostingRegressor`, un modelo basado en árboles con boosting, adecuado para variables tabulares y patrones no lineales.

La séptima fase es la evaluación. Se calculan métricas como MAE, RMSE y R2. El MAE mide el error absoluto medio y es fácilmente interpretable en euros por megavatio hora. El RMSE penaliza más los errores grandes, por lo que resulta útil para detectar predicciones especialmente desviadas. El R2 mide la proporción de variabilidad explicada por el modelo. Además, se compara el rendimiento contra una baseline sencilla denominada `lag 24`, que consiste en usar como predicción el precio observado 24 periodos antes.

La última fase es la inferencia. Una vez entrenado el modelo, se serializa en disco mediante `joblib`. Posteriormente, el sistema puede cargar ese modelo y generar una predicción para el siguiente periodo disponible después del último dato del dataset. Esta funcionalidad permite completar el ciclo de trabajo desde la descarga de datos hasta la generación de una estimación futura.

La metodología se apoya en un enfoque incremental. Primero se construye un modelo funcional usando únicamente información histórica del precio. Después se añaden modelos alternativos y un modo automático de selección. Finalmente se incorpora la posibilidad de usar datos externos de generación renovable. Esta evolución permite analizar progresivamente qué aporta cada mejora y evita convertir el sistema en una caja negra demasiado compleja desde el inicio.

Otra decisión metodológica importante es mantener una baseline sencilla. En este caso se utiliza `lag 24`, que representa el precio observado 24 periodos antes. Esta referencia es deliberadamente simple, pero muy útil: si un modelo complejo no supera a una regla tan básica, su utilidad práctica queda en duda. La presencia de una baseline obliga a interpretar las métricas en contexto y no solo en términos absolutos.

El trabajo también contempla la generación de gráficas de validación. Estas gráficas permiten comparar visualmente la serie real, la predicción del modelo y la baseline. Aunque las métricas numéricas son imprescindibles, la visualización ayuda a detectar comportamientos que pueden pasar desapercibidos: retrasos en la predicción, suavizado excesivo, incapacidad para capturar picos o errores sistemáticos en determinados tramos.

En relación con la integración de ESIOS, la metodología adopta una aproximación prudente. Las variables renovables se incorporan como opción, no como requisito obligatorio. De este modo, el proyecto puede ejecutarse sin token ESIOS y sigue siendo funcional. Si se dispone de token, el usuario puede activar la descarga de previsiones y enriquecer el dataset. Esta decisión mejora la accesibilidad del proyecto y permite comparar explícitamente el rendimiento con y sin variables externas.

### 1.4. Resultados esperados

El resultado esperado del trabajo no es obtener una predicción perfecta del precio eléctrico, ya que se trata de un problema complejo y sujeto a múltiples factores externos. El objetivo es construir un sistema funcional, reproducible y extensible que permita evaluar distintos modelos y observar como cambia su comportamiento al modificar datos, variables y rangos temporales.

Se espera que los modelos entrenados superen a una baseline simple basada en el precio del día anterior. Esta baseline es relevante porque el precio eléctrico presenta patrones diarios y semanales. Sin embargo, limitarse a repetir el precio de 24 periodos antes no captura cambios de tendencia, episodios de volatilidad o variaciones en la generación renovable. Un modelo que incorpore retardos, medias móviles y variables de calendario debería ser capaz de mejorar esta referencia.

En experimentos iniciales con datos de enero a marzo de 2025, el modelo `RidgeCV` obtuvo el mejor comportamiento entre los candidatos evaluados. En cambio, al ampliar el entrenamiento a prácticamente todo el año 2025, el modelo `HistGradientBoostingRegressor` mostró un rendimiento superior. Este resultado ilustra un aspecto importante: el modelo más adecuado puede depender del periodo analizado, de la cantidad de datos disponible y de las características del mercado durante ese intervalo.

También se espera que la incorporación de variables ESIOS pueda mejorar la capacidad predictiva en determinados escenarios. La producción renovable, especialmente solar y eólica, tiene un impacto notable en el mercado eléctrico. Por ejemplo, una alta previsión solar durante horas centrales del día puede coincidir con precios más bajos, mientras que una baja generación renovable puede favorecer precios más elevados. No obstante, el impacto real debe verificarse experimentalmente, ya que la mejora dependerá de la calidad, disponibilidad y alineación temporal de los datos externos.

Otro resultado esperado es disponer de una herramienta práctica que facilite la experimentación. La interfaz gráfica permite ejecutar las fases del flujo sin necesidad de recordar comandos. El usuario puede seleccionar el rango de fechas, elegir modelo, activar o no ESIOS y observar el progreso mediante logs. Esta interfaz refuerza el carácter aplicado del trabajo y permite presentar el proyecto de forma más clara.

Desde el punto de vista documental, el trabajo debe proporcionar una base completa para explicar el problema, justificar las decisiones técnicas y analizar los resultados obtenidos. La aplicación práctica no se plantea como un producto cerrado, sino como un prototipo funcional sobre el que se pueden proponer mejoras futuras.

Se espera que los resultados permitan extraer varias conclusiones. La primera es que la cantidad de histórico disponible afecta notablemente al rendimiento. Entrenar con pocos meses puede favorecer modelos distintos a los que resultan mejores con un año completo. La segunda es que las variables de ingeniería temporal tienen un papel fundamental. Incluso modelos relativamente sencillos pueden obtener buenos resultados si las variables capturan adecuadamente la estructura diaria y semanal del precio. La tercera es que el uso de variables externas debe evaluarse empíricamente: aunque tenga sentido teórico, su valor dependerá de la calidad y disponibilidad de los datos.

El trabajo también puede mostrar que no existe un único modelo universalmente superior. En una etapa inicial, una regresión regularizada puede comportarse mejor que una red neuronal, especialmente si el dataset no es muy grande. Con más datos, un modelo de boosting puede capturar patrones más complejos. Esta observación es importante porque evita una conclusión simplista basada en la idea de que los modelos más complejos son siempre mejores.

Otro resultado esperado es identificar errores concretos y analizarlos. Por ejemplo, una predicción para un festivo como el 1 de enero puede presentar desviaciones significativas si el modelo no dispone de variables que indiquen festivos o cambios especiales de demanda. Estos casos son valiosos porque revelan limitaciones del enfoque y orientan futuras mejoras. En un Proyecto, el análisis de errores puede ser tan importante como la mejora de las métricas globales.

En términos de aplicación, se espera que el usuario pueda ejecutar el proyecto completo sin modificar código fuente. La interfaz gráfica debe permitir realizar las operaciones principales, mientras que la interfaz por consola debe facilitar experimentos repetibles. Esta doble modalidad permite que el proyecto sea útil tanto para demostraciones visuales como para ejecuciones técnicas controladas.

### 1.5. Aplicación práctica desarrollada

La aplicación práctica desarrollada recibe el nombre de `omie_v2_python`. Se trata de un proyecto Python organizado como paquete instalable, con comandos de consola y una interfaz gráfica. El paquete principal se denomina `omie_price_nn` y contiene los módulos necesarios para descargar datos, generar variables, entrenar modelos, realizar inferencia e integrar datos externos de ESIOS.

El proyecto se ejecuta dentro de un entorno virtual `.venv`, lo que permite aislar dependencias y facilitar la instalación. Las dependencias principales son `pandas`, `numpy`, `requests`, `scikit-learn`, `joblib` y `matplotlib`. Estas bibliotecas permiten cubrir las necesidades principales del flujo: manipulación de datos, descarga HTTP, entrenamiento de modelos, serialización y visualización.

La estructura del proyecto separa claramente los datos, los modelos, el código fuente y la documentación. Los ficheros descargados de OMIE se almacenan en `data/raw/`, mientras que los datasets procesados se guardan en `data/processed/`. Los modelos entrenados y las gráficas de validación se almacenan en `models/`. La documentación se centraliza en el directorio `documents/`, donde se incluyen documentos relacionados con el Proyecto y la descripción del sistema.

Desde consola, la aplicación ofrece comandos como `omie-price-train`, `omie-price-predict` y `omie-price-gui`. El primero permite entrenar modelos indicando fecha inicial, fecha final y modelo seleccionado. El segundo carga el modelo guardado y realiza inferencia. El tercero abre una interfaz gráfica desarrollada con Tkinter.

La interfaz gráfica muestra las fases principales del flujo: extracción, preparación, entrenamiento y test, inferencia y apertura de gráfica. También incluye un selector de modelo con cuatro opciones: `auto`, `ridge`, `mlp` e `hist_gradient_boosting`. El modo `auto` entrena todos los modelos disponibles y guarda el de menor MAE. Esta opción resulta adecuada para usuarios que desean comparar modelos sin ejecutar manualmente cada uno.

La integración con ESIOS se ha implementado de manera opcional. Si el usuario dispone de un token de la API, puede activar la casilla ESIOS e introducir el token en la interfaz gráfica. Alternativamente, puede configurar la variable de entorno `ESIOS_TOKEN` y ejecutar el entrenamiento desde consola con la opción `--include-esios`. Si se activa esta funcionalidad, el sistema descarga previsiones de generación eólica y solar, las une al dataset de OMIE y las incorpora como variables adicionales de entrenamiento.

La aplicación también genera una gráfica de validación en `models/validation_plot.png`. Esta gráfica compara el precio real, la predicción del modelo y la baseline `lag 24`. De este modo, el usuario puede observar visualmente si el modelo sigue la tendencia del mercado y si mejora respecto a una estrategia simple basada en el precio del día anterior.

En conjunto, la aplicación práctica demuestra un flujo completo de ciencia de datos aplicado al mercado eléctrico: adquisición de datos, procesamiento, ingeniería de variables, entrenamiento, validación, selección de modelo, inferencia y presentación de resultados. Esta herramienta constituye la base experimental del Proyecto y permite desarrollar futuros capítulos con resultados cuantitativos, análisis de errores y propuestas de mejora.

La aplicación ha sido disenada para que el usuario pueda comprender el proceso por fases. La fase de extracción se encarga de obtener los datos brutos. La fase de preparación transforma esos datos en un formato apto para entrenamiento. La fase de entrenamiento compara modelos o ejecuta el modelo seleccionado. La fase de inferencia genera una predicción. Esta división facilita la depuración y permite explicar con claridad qué ocurre en cada etapa.

El proyecto también incorpora control de versiones mediante Git. Esto permite registrar la evolución del código y de la documentación. Durante el desarrollo se han creado commits para hitos concretos, como la incorporación de la interfaz gráfica, la selección de modelos, la integración de ESIOS y la ampliación de la documentación. Esta práctica es relevante en un Proyecto técnico, ya que refleja una metodología de trabajo ordenada y trazable.

Aunque la aplicación práctica ya permite entrenar modelos y realizar inferencias, se considera un punto de partida. La arquitectura propuesta permite incorporar nuevas funcionalidades, como predicción de todo el día siguiente, backtesting mensual, optimización automática de hiperparámetros o inclusión de variables meteorológicas. Estas mejoras se recogen en las líneas futuras del trabajo.

### 1.6. Contribuciones del trabajo

El trabajo aporta varias contribuciones concretas. La primera es la construcción de un flujo completo de predicción del precio eléctrico a partir de datos públicos. No se limita a entrenar un modelo aislado, sino que cubre todo el ciclo de vida básico de un proyecto de aprendizaje automático: adquisición de datos, almacenamiento, limpieza, generación de variables, entrenamiento, evaluación, inferencia y documentación.

La segunda contribución es la comparación de modelos de distinta naturaleza bajo una misma estructura experimental. La regresión Ridge representa un enfoque lineal regularizado; la red neuronal MLP representa un enfoque conexiónista sencillo; y el gradient boosting representa un enfoque basado en árboles. Esta comparación permite estudiar diferencias de rendimiento, coste computacional e interpretabilidad.

La tercera contribución es la integración opcional de datos de generación renovable. La inclusión de previsiones eólicas y solares conecta el modelo con factores físicos del sistema eléctrico, no solo con patrones estadísticos del precio pasado. Esta extensión aproxima el proyecto a un enfoque más realista, ya que la generación renovable es una de las variables más influyentes en el mercado actual.

La cuarta contribución es la creación de una interfaz gráfica. Aunque el trabajo podría haberse limitado a scripts de consola, la GUI permite visualizar el flujo de manera más accesible. Esto resulta útil tanto para usuarios sin experiencia en consola como para la defensa del Proyecto, donde una demostración visual puede ayudar a explicar el funcionamiento del sistema.

La quinta contribución es la documentación estructurada del proyecto. El repositorio incluye un README detallado y un directorio específico para documentos académicos. Esta documentación facilita que otra persona pueda instalar, ejecutar y comprender el sistema. Además, proporciona la base para desarrollar los capítulos posteriores del Proyecto.

### 1.8. Síntesis del capítulo

La predicción del precio eléctrico español es un problema complejo, relevante y adecuado para aplicar técnicas de aprendizaje automático. El proyecto desarrollado combina modelos lineales, redes neuronales y modelos de boosting. Además, incorpora una baseline sencilla para evaluar si los modelos aportan valor real. La aplicación práctica `omie_v2_python` materializa esta metodología en un software ejecutable. Permite descargar datos, preparar el dataset, entrenar modelos, comparar métricas y realizar inferencia. Su interfaz gráfica facilita la demostración del flujo completo, mientras que los comandos de consola permiten reproducir experimentos de forma controlada.

En definitiva, este Proyecto propone una aproximación aplicada a la predicción del mercado eléctrico español, integrando conocimientos de energía, programación, análisis de datos y aprendizaje automático.
